WEB SERVICE
ActionMap

Actionmap is an app that allows users to learn more about their representatives, political events in their area as well as aggregate, share and view news items in their locality.

In an Agile development environment, I developed and added functionality to an existing RESTful web service/SaaS application using Model, View, Controller (MVC) architectural pattern and SOLID principles.

Following Agile methodologies, my 4-student engineering team collaborated during daily scrums and retrospectives. We also collaborated in writing user stories using Behavior-Driven Development (BDD) and prioritizing/assigning points. I kept track of user story delivery using Pivotal Tracker. I also enabled Pivotal Tracker integration with GitHub to use branch-per-feature development methodology.

I worked with client-side JavaScript code as well as Rails. Since ActionMap relies heavily on JavaScript, I used Webpack to manage JavaScript assets and Bundler to manage Ruby on Rails assets. (Webpack is a sophisticated JavaScript library manager that relies on Node.js and other tools that I will install.)

ActionMap uses the Google Civic Info API. I signed up for a developer API key, which are sensitive. Due to the sensitive data that ActionMap handles, I securely managed this data by upgrading the software's security. I set up single sign-on third-party authentication using OAath2.

This development process also included:

- Creating Cucumber tests
- Continuous integration using Travis CI
- Consistently running code linters to check our code for code smells and bad habits
- Consistently measuring test coverage with CodeCov
- Performing PostgreSQL database queries using Rails Active Record
- Using branch-per-feature and pull-request-based code reviews

Languages/Frameworks: Ruby on Rails, HTML, CSS, JavaScript, jQuery, Webpack (Node.js), Bootstrap, ActiveRecord (PostgreSQL database system), HAML, OAath2, Cucumber, RSpec

Technologies: Travis CI, Pivotal Tracker, Heroku

WEB SERVICE
Rotten Potatoes!

I upgraded an existing database-backed (PostgreSQL) web service with new features: "User can include name of Director with a movie", and "Given a movie with a director, user can search for other movies with same director".

I use TDD to develop these features and achieve 100% statement coverage (C0) of RSpec tests for the features. I integrated a combination of acceptance and unit tests. Specifically, Cucumber scenarios to test new feature, and unit tests with RSpec so that the Cucumber scenarios pass.

This app also uses the The Movie Database (TMDB) API. I performed SQL database queries using Rails' Active Record.

After this project, I am able to do the following:

- Create a RESTful route in order to find similar movies.

- Create and run simple Cucumber scenarios to test a new feature.

- Use RSpec to create unit and functional tests that drive the creation of app code that lets the Cucumber scenario pass.

- Understand the MVC design pattern: I understand where to modify a Rails app to implement the various parts of a new feature, since a new feature often touches the database schema, model(s), view(s), and controller(s).

- Use continuous integration with Travis to continuously monitor test coverage and passing status.

Languages/Frameworks: HTML, CSS, JavaScript, jQuery, Bootstrap, Ruby on Rails, ActiveRecord, Cucumber, RSpec, HAML

Technologies: Travis CI, Pivotal Tracker, Heroku
